# FEX Sonic Compatibility Slice Design

## Goal

Run the legally supplied Sonic Mania title `CUSA07023` through a native AArch64
shadPS4 host whose PS4 x86-64 guest instructions execute in FEXCore. Acceptance
is first-level completion with working rendering, audio, and controller input,
followed by ten uninterrupted minutes on the SM8650 tablet.

This slice does not make FEX the global default. Box64 remains the current,
separate fallback until every acceptance gate below has passed on device.

## Verified Starting Point

Phase 1 is complete at `53202d93`: the packaged, native-AArch64 FEXCore harness
creates contexts and threads, executes controlled x86-64 code, crosses a narrow
host bridge, invalidates code, and passes the SM8650 evidence gate. It is not
connected to shadPS4 game execution.

The current runtime launches an x86-64 `shadps4` binary under Box64. The native
execution assumptions which an ARM64 build cannot retain are explicit:

- `src/core/module.cpp` casts a loaded PS4 initializer address to `EntryFunc`
  and invokes it directly.
- `src/core/linker.cpp` starts the main entry with `RunMainEntry`.
- `src/core/libraries/kernel/threads/pthread.cpp` has an ARM64-unimplemented
  guest-stack entry helper and starts guest pthread routines directly.
- `src/core/libraries/libs.h` registers each HLE function as only a raw native
  host pointer. Relocation writes that pointer into PS4 memory.
- `src/core/fex/fex_guest_engine.*` only owns a synthetic mapped page and a
  synthetic `syscall` bridge; it has no module mapping, real HLE, or title
  thread integration.

Therefore an ARM64 compile alone, or running the whole x86 shadPS4 process in
FEX, cannot meet this goal.

## Decision

Use a native ARM64 shadPS4 executable with an explicit guest-CPU boundary and a
FEX implementation of that boundary. Do not use FEXInterpreter, Wine,
ARM64EC, an external x86 process, or a Box64 fallback hidden behind FEX mode.

The boundary has four responsibilities:

1. Create and destroy one FEX context and one FEX thread state on each owning
   native host thread.
2. Execute PS4 code at a mapped guest RIP with a mapped guest RSP and exact
   GPR/RFLAGS/XMM state.
3. Resolve guest imports to generated x86-64 FEX call veneers, not AArch64
   host addresses. A veneer enters the FEX syscall handler, which invokes the
   registered native HLE implementation through a typed ABI adapter and returns
   to guest code.
4. Enter guest code for main entry, dynamic-module init, guest pthread starts,
   and native-HLE-to-guest callbacks without a C++ cast from guest address to
   host function pointer.

The host and guest share shadPS4's fixed virtual PS4 mappings. Guest pointers
therefore remain valid host pointers only after mapping/range validation; this
does not permit guest code to receive arbitrary ARM host addresses.

## HLE ABI Model

`SymbolsResolver` will keep function records separately from object records.
Each function record contains the native host target and a type-erased invoker
generated by the existing `LIB_FUNCTION` registration site. The generated
invoker decodes the PS4 SysV integer, vector, and stack argument locations from
the FEX CPU frame, calls the registered AArch64 HLE implementation, and writes
integer/vector return state back to the same frame. Unsupported signatures are
reported with symbol identity and a precise error; they are never called via an
untyped cast.

`FexGuestCpu` allocates a small read/execute veneer pool inside guest memory.
Each function relocation receives a veneer address. The veneer identifies the
registered symbol and executes the FEX-supported bridge instruction; its return
uses normal guest x86-64 control flow. Object relocations continue to resolve
only to valid guest-visible data mappings.

This is reusable FEX embedding logic. Wine loader/rootfs, ARM64EC, Linux guest
syscalls, and GameNative-specific thunks are excluded.

## Thread, Fault, and Memory Rules

- A `GuestThread` owns an `InternalThreadState`, mapped guest stack, initial
  CPU frame, and shutdown state. It is created and destroyed on its host
  thread; cross-thread use is an error.
- Main entry, dynamic module initialization, and pthread entry use the same
  `GuestThread::Run` path. Native HLE callbacks use an explicit nested guest
  call API with a guest return sentinel; no re-entrant C++ function-pointer
  cast is allowed.
- FEX signal delegation remains installed for FEX threads. shadPS4 access and
  illegal-instruction handling must chain through the supported delegator path,
  not replace it globally.
- The backend validates all mapped guest ranges, preserves W^X, checks the
  device page size, flushes code after veneer writes, and invalidates FEX code
  after executable-memory mutation.
- Failure is a typed result containing stage, errno/FEX detail, guest RIP when
  safe, and symbol ID where relevant. It must stop the FEX attempt; it may not
  silently launch Box64.

## Delivery Sequence

1. Establish source contracts and compile a native ARM64 shadPS4 target while
   retaining the current x86/Box64 runtime unchanged.
2. Add the guest-CPU interface, real mapped main-entry execution, and the
   smallest typed HLE veneer/dispatcher set proven by Sonic's import log.
3. Add guest pthreads and callback nesting, then validate boot-to-menu and the
   first rendered/audio/controller frame using an explicit per-title FEX
   selection for `CUSA07023`.
4. Qualify first-level completion and a ten-minute stability run. Only after
   this proof may the follow-up change make FEX the default for all games.

## Non-Goals

- No global FEX default, Box64 removal, Bloodborne performance claim, game-data
  extraction, or unbounded “all HLE functions” rewrite.
- No fake HLE stubs, ignored failures, broad error swallowing, global SELinux
  or executable-memory bypass, or changes to GameNative or the legacy worktree.
- No use of the historical `feature/fex-arm64-sonic` branch as an implementation
  source. It contains Phase 0 probe/evidence work but no validated real-game
  backend.

## Evidence Required

Automated contracts must prove that FEX mode never writes an ARM host function
address into a function relocation, rejects an unsupported ABI signature, maps
and tears down per-thread state in order, and has no Box64 launch fallback.
Native ARM64 artifact checks must prove `EM_AARCH64` for the new shadPS4 binary
and FEXCore dependencies. Device evidence must record sanitized backend ID,
FEX revision, title ID, launch stages, first video/audio/input success,
first-level result, ten-minute duration, and clean exit/crash status without
serials, paths, game content, or memory dumps.
